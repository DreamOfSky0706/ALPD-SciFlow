# 节点式图像处理工作流工具 —— SciFlow

> 一个基于 Qt 及其若干衍生库的科研制图 × 数据处理 × 图像处理的一站式节点图像处理工具。

---

## 项目简介

SciFlow 是一个基于节点式操作方式的桌面端图像处理与数据可视化工具。用户通过拖拽节点、连接端口、配置参数的方式搭建处理流水线，一键执行并导出结果。工作流可以保存为 JSON 文件反复使用，也支持命令行批处理模式。

项目涵盖原始图像处理、子图排版、数据可视化、流程图绘制、学科装饰图案生成等真实科研工作流场景。

### 特点

- **135 种节点**，覆盖 17 大类：图像读写、几何变换、色彩调整、滤波模糊、形态学、边缘检测、通道操作、图层合成、文字标注、数据读写、数据处理、图表生成、风格化特效、版面排版、图形绘制、流程控制、几何图案
- **5 种数据类型**：Image / ImageList / DataTable / Numeric / Text，带颜色编码的端口和连线
- **交互优良的 GUI**：基于 Qt 6 的完整图形界面，含画布编辑、属性面板、悬浮参数编辑、预览、日志、工具箱搜索、收藏夹、拖拽连线、框选批量操作，交互符合直觉
- **工作流持久化**：JSON 格式保存/加载，含版本控制与兼容性检查
- **命令行模式**：`--no-gui --workflow xxx.flow.json` 批量执行，支持参数覆盖
- **3 个内置模板**：论文四子图排版、批量水印、数据分析制图
- **28 组测试用例** + **7 组展示用例**：覆盖功能全面

## 快速开始

### 环境要求

- Windows 10/11 (x64)
- Visual Studio 2026
- Qt 6.11.1 (MSVC 2022 64-bit)
- OpenCV 4.12.0
- CMake 3.20+

### 构建步骤

```powershell
# 1. 解压项目到 D:\SciFlow（例）

# 2. 配置 CMake（请根据实际安装路径调整 Qt 和 OpenCV 目录）
cmake -B build -G "Visual Studio 18 2026" -A x64 `
  -DQt6_DIR="D:/Qt/6.11.1/msvc2022_64/lib/cmake/Qt6" `
  -DOpenCV_DIR="D:/opencv/build/x64/vc16/lib"

# 3. 编译
cmake --build build --config Debug

# 4. 运行 GUI 模式
.\build\Debug\ImageNodeTool.exe

# 5. 运行 CLI 模式
.\build\Debug\ImageNodeTool.exe --no-gui --workflow test_workflows\test_02.flow.json
```

**QCustomPlot** 和 **QXlsx** 已包含在 `third_party/` 目录中，无需额外安装，**仅需保证已有且正确配置 Qt 和 OpenCV 即可**。

## ！！！！！给助教的特别说明！！！！！

### 1. .slnx 无法直接编译！！！

项目根目录下的 `.slnx` 和 `.vcxproj` 是 CMake 生成的构建文件，其中引用了自动生成的文件（`mocs_compilation_*.cpp`、`qrc_resources.cpp` 等），且包含对 Qt 和 OpenCV 的**绝对路径引用**。因此：

- **直接双击 .slnx 打开 VS 后按 F5 无法编译，这些文件（包括项目同名文件夹）是我面向作业要求后建的，没有任何意义**，必须先通过 CMake 配置生成正确的构建文件
- 请按照上方构建步骤中的 CMake 命令，根据你本机的 Qt 和 OpenCV 安装路径重新配置
- 如果需要直接运行，请使用 `build/Debug/ImageNodeTool.exe`（预编译二进制，需同目录下有 OpenCV DLL）

### 2. 测试用例与展示用例

- `test_workflows/` 目录含 28 个 `.flow.json` 测试工作流，集中于单点测试
- `showcases/` 目录含 7 组展示用例，侧重于综合展示
- 部分测试需要桌面上的 CSV 数据文件（`test_data.csv` 等），或自行修改读取相关的节点
- 部分测试需要桌面上的测试图片（`test_input.png`）和 `test_batch_input/` 目录，或自行修改读取相关的节点

## 最后一舞

本次作业中，**我没有亲手写一行代码**。

对我来说，这次作业的得分如何已经不是我关注的议题了（否则我会老老实实地在 VS 里一个个 cpp 写过去，完成一个不出彩也不出错的作业，稳稳地拿到最后一次的分数），我更想借这个作业的机会做两件事情。

其一，是做一个真正创新且有用的东西。这一点也请详见 `需求.md`，和之前九个作业不同的是，这一次，我真的做到了自己满意。我爱这份作品，我认可这份作品。

其二，是做一个实验。

在传统软件工程中，需求文档往往是“仅供参考”的，而真正决定代码，甚至是最终成品的，是程序员对需求的理解、翻译和即兴发挥。但假如需求文档精确到每一个参数的类型、范围、默认值，精确到每一个端口的名称和数据类型，精确到每一种异常的处理方式，精确到每个架构细节，那么大模型，这个从底子上就带着高度不确定性的东西，是否真的能一致地输出我想要的东西？

于是我做了以下工作：

1. **亲自设计了全部架构**。从选题到落实，从文件拆分到框架设计，从模块划分到类层次结构，从工具依赖到具体实现，全部由我设计。

2. **亲自定义了全部 135 种节点**。每种节点的输入端口、输出端口、参数列表（名称、控件类型、取值范围、默认值、条件可见性）、处理逻辑（包括边界条件和异常处理），我都逐一编写在需求文档中。这份需求文档约四万字，我确信其精确程度已经达到了“任何开发者照着实现，都能产出完全一致的工具”。

3. **亲自设计了测试体系**。28 组测试用例覆盖 79 种节点类型，7 组展示用例覆盖全部应用场景。每个测试用例的输入数据都是我准备的，每个 bug 都是我逐个发现并指导修复的。

4. **代码由 AI 生成，但设计决策始终在我**。我审核了每一处架构决策，拒绝了不合理的建议，坚持了关键的设计原则。大模型的角色是按照我的文档，准确、高效地写出代码，我在尽可能地让它回归它本应该成为的角色。

总的来说，我做的工作是除了写代码本身这一步外的其他所有。

这是一个非常大胆的尝试，但它也有明显的局限性，比如最显而易见的，依赖特定的 Qt/OpenCV 版本和绝对路径，需要人工配置环境；大规模项目超出了单次对话的上下文窗口，需要人工维护状态；AI 生成的代码在性能优化和边界情况处理上仍需要人工把关。

但无论如何，我对自己的这次尝试很满足，即便深知自己可能会因此付出一定的代价。如果读到这里的你愿意的话，我也想邀请你参与进来，成为这个实验的一部分——将 `需求.md` 扔给性能较强的大模型，看看得到的成品与我的版本是否高度一致。

感谢这门课程给我这个自由探索的空间。
